# Runbook: перезапуск зависших workflow в n8n

## Когда считать workflow зависшим
Workflow считается зависшим, если:
- выполнение в статусе `running` более ожидаемого SLA (например, > 10 минут для inbound-lead);
- в queue mode execution не движется и отсутствуют новые логи шагов;
- есть накопление задач в очереди без воркеров.

## Быстрый triage (3–5 минут)
1. Откройте **Executions** и найдите застрявший execution.
2. Зафиксируйте:
   - `workflow name`, `execution id`, `started at`, последний успешно выполненный узел.
3. Проверьте инфраструктуру:
   - доступность API зависимых систем (CRM/Alert/Audit/DLQ);
   - состояние БД n8n;
   - состояние воркеров (если queue mode).

## Безопасный перезапуск execution
1. Остановите конкретный execution через UI (**Stop execution**).
2. Если есть частично обработанный payload, убедитесь, что повтор безопасен:
   - либо endpoint идемпотентный;
   - либо есть уникальный `leadId`/`messageId` в payload.
3. Нажмите **Retry from failed node** (предпочтительно) либо **Execute workflow** с тем же тестовым payload.

## Массовое застревание (несколько execution)
1. Деактивируйте проблемный workflow временно.
2. Устраните причину (credentials, API downtime, лимиты).
3. Перезапустите n8n worker/process:
   - Docker: `docker restart <n8n-worker-or-main-container>`
   - Systemd: `systemctl restart n8n`
4. Активируйте workflow обратно.
5. Запустите reprocessing из DLQ по записям за период инцидента.

## Работа с DLQ
1. Найдите записи DLQ по `workflow` и временному окну.
2. Отфильтруйте дубликаты по уникальному ключу (`leadId`, `reminderId`, `reportDate`).
3. Реиграйте события батчами (рекомендуется 20–50 элементов).
4. Контролируйте алерты, чтобы избежать алерт-штормов.

## Проверки после восстановления
- Inbound Lead: POST webhook возвращает 202, audit записывается, escalation работает по условию.
- Reminders: cron срабатывает каждые 15 минут, отправки идут без роста DLQ.
- Daily Report: в 20:00 отчёт отправляется и логируется.

## Эскалация инцидента
Эскалируйте инцидент, если:
- повторный перезапуск не помогает;
- деградация > 30 минут;
- потеря/дубль бизнес-событий подтверждены.

Минимальные данные для эскалации:
- период инцидента (UTC),
- affected workflows,
- количество затронутых execution,
- примеры `execution id`,
- ссылка на дашборд/логи.
